# Rust 代码警告处理指南

## 处理未使用代码警告的几种方式

### 方式 1: Crate 级别统一设置（推荐）✅

在 `main.rs` 或 `lib.rs` 的顶部添加：

```rust
#![allow(dead_code)]
```

**优点：**

- 一次性解决所有警告
- 代码更简洁
- 适合库代码（这些函数是 API 的一部分）

**缺点：**

- 可能会隐藏真正需要清理的代码

### 方式 2: 模块级别设置

在模块顶部添加：

```rust
#[allow(dead_code)]
mod some_module;
```

### 方式 3: 函数/结构体级别设置（当前方式）

在每个未使用的项上添加：

```rust
#[allow(dead_code)]
pub fn some_function() { }
```

**优点：**

- 精确控制
- 明确标记哪些是预留的 API

**缺点：**

- 代码较冗长

### 方式 4: 删除未使用的代码

如果确定不会使用，直接删除。

**适用场景：**

- 确定不会使用的代码
- 临时测试代码

### 方式 5: 条件编译

使用 `#[cfg]` 属性：

```rust
#[cfg(feature = "biometric")]
pub fn biometric_function() { }
```

**适用场景：**

- 可选功能
- 平台特定代码

## 当前项目的选择

我们选择**方式 1（Crate 级别设置）**，因为：

1. **这些是库 API**：密码强度检测、BNB 链支持等功能是库的一部分，虽然当前未使用，但将来可能会用到
2. **保持 API 完整性**：保留这些函数可以让其他模块或未来功能使用
3. **代码更简洁**：不需要在每个函数上添加属性

## 建议

- ✅ **库代码**：使用 crate 级别 `#![allow(dead_code)]`
- ✅ **应用代码**：删除未使用的代码或使用函数级别设置
- ✅ **可选功能**：使用条件编译 `#[cfg(feature = "...")]`
